# 项目协作规范

在本项目中生成、修改或调试代码时，遵循以下要求。

## 1. 开始任务

1. 先阅读用户指定的文件和相关代码，不要仅根据文件名猜测实现。
2. 用一句话复述任务目标，并明确本次工作的边界。
3. 若需求存在会影响实现方向的歧义，先向用户提问；不因小问题中断工作，可采用合理默认值并记录。

## 2. 过程控制

1. 将任务拆成可验证的小步骤，避免一次进行大范围、无关的重构。
2. 修改前确认现有接口、配置、张量形状、数据类型、设备和分布式行为。
3. 不擅自删除已有功能，不修改与当前任务无关的文件。
4. 涉及训练、量化、分布式或集群任务时，优先保留断点恢复、随机种子、错误处理和资源清理能力。
5. 命令可能消耗大量 GPU、覆盖文件、删除数据或影响正在运行的任务时，执行前必须说明风险并征得用户同意。

## 3. 主动提出难点和不确定项

遇到以下情况时，不得假装确定：

- 无法从代码或文档确认的需求；
- 存在多种实现且会影响结果、性能或兼容性；
- 张量维度、数据格式、模型接口或配置含义不明确；
- 缺少依赖、数据、权限、环境信息或集群参数；
- 修改可能破坏 checkpoint 兼容性、数值精度或分布式一致性；
- 测试未覆盖，或当前环境无法完成验证。

使用以下格式向用户说明：

```markdown
### 待确认
- 问题：
- 已知信息：
- 为什么重要：
- 建议方案：
- 可选方案：
```

能够通过阅读代码、运行低成本检查或查阅项目文档确认的问题，应先自行调查，再向用户提问。

## 4. 代码与操作记录

每次生成或修改代码后，将记录追加到 `.claude/task.md`，不要覆盖已有记录。格式如下：

```markdown
## YYYY-MM-DD HH:MM - 任务名称

### 目标
- 本次任务要解决的问题。

### 修改
- `文件路径`：修改内容及原因。

### 关键决策
- 采用的方案、默认值及理由。

### 难点与不确定项
- 已解决的问题及结论。
- 尚未确认的问题、风险及需要用户决定的事项。

### 验证
- 已运行的检查、测试或命令及其结果。
- 未执行的验证及原因。

### 后续
- 必要的下一步；没有则写“无”。
```

记录必须反映实际完成的工作，不得把计划写成已完成。命令、测试或实验失败时，也要记录失败原因和当前状态。

## 5. 日志要求

新增可执行脚本、训练入口或长时间运行任务时：

1. 使用项目现有日志框架；没有统一框架时，Python 使用 `logging`，不要只使用 `print`。
2. 日志至少包含启动配置、关键阶段、进度、输出路径、异常和结束状态。
3. 不在日志中写入 token、密码、私钥或其他敏感信息。
4. 高频循环应控制日志频率，避免显著影响性能或生成过大的日志文件。
5. 分布式任务默认仅由主进程输出全局日志；各 rank 的错误信息应保留必要上下文。

## 6. 完成任务

结束前必须：

1. 检查改动是否符合任务范围；
2. 运行与风险相匹配的测试或静态检查；
3. 更新 `.claude/task.md`；
4. 向用户简要说明已完成内容、验证结果和仍存在的不确定项。

不得声称未运行的测试已经通过，也不得隐瞒无法验证的部分。
